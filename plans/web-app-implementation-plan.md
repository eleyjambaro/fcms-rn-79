## 1. Context

FCMS (Food Cost Management System) is an offline-first React Native POS/inventory app (`/Users/eleyjambaro/fcms-rn-79`). The user wants a **production-grade, scalable, maintainable, responsive web version** with the **same logic, features, and functionality**, deployable on **Vercel**, using **Next.js (latest stable), shadcn/ui + Tailwind, TanStack Query**, backed by the existing Laravel API (`/Users/eleyjambaro/fcms-api`).

**The defining architectural problem:** `fcms-api` exposes **only** auth, device/branch/account/role/license management, and a **delta-sync push/pull protocol** — it has **no REST CRUD** for business data. All business logic (cost calc, reports, inventory valuation, batch transfer, sales) currently lives **client-side in the RN app's SQLite layer** (`/Users/eleyjambaro/fcms-rn-79/src/localDbQueries/`, ~33 files). The data itself *does* live server-side in PostgreSQL (the `Models/Sync/*` tables, populated by sync), but is only reachable via the sync protocol.

**Approved approach (user decisions):**
1. **Extend `fcms-api` with a REST + server-side business-logic layer** over the existing sync tables. Web writes follow the **sync invariants** so mobile devices stay consistent. Next.js is a thin client.
2. **Full feature parity, phased internally** — plan everything; build sequentially; no separate early-MVP release gate.
3. **BFF (backend-for-frontend) auth**: Next.js Route Handlers proxy all API calls server-to-server; the Sanctum bearer token lives in an **httpOnly cookie** (never in browser JS). Solves CORS, best security, Vercel-native.
4. **Web logical device + branch switcher**: the web app registers as one logical device per company; users switch the active branch via a header dropdown, gated by branch-account assignments.

**Intended outcome:** A new sibling repo `/Users/eleyjambaro/fcms-cloud-nextjs` (a Next.js app) plus additive REST endpoints in `fcms-api`, delivering the full FCMS feature set on the web, kept consistent with mobile through the shared sync tables.

## 2. Why these decisions (rationale, kept for future sessions)

- **REST + server logic over browser-SQLite-sync:** A browser SQLite + sync-engine port would pull an entire branch's dataset into the browser (poor scale), and re-implement the whole sync engine — not maintainable for web. Putting CRUD + aggregate reports next to the data (PostgreSQL/Laravel) is the scalable, cacheable, Vercel-friendly choice.
- **Web writes obey sync invariants → mobile↔web consistency for free.** Because all sync entities relate by `*_sync_id` and carry `sync_id/company_id/branch_id/device_id/updated_at/is_deleted`, a REST write that follows the same conventions becomes a normal row that mobile pulls on its next sync (the web device's `device_id` differs from any mobile device, so echo-suppression doesn't hide it). This is the linchpin — see §5.1.
- **BFF + httpOnly cookie:** keeps the Sanctum token off `window`, removes the need to enable CORS on the Laravel API for the browser (browser only talks same-origin to Next.js), and maps cleanly to Vercel serverless route handlers.
- **Web logical device + branch switcher:** "one physical device, one branch" is a mobile concept; a desktop dashboard user expects to switch branches freely. Access is still controlled by `branch-account-assignments` + role.

## 3. High-level architecture

```
Browser (React, shadcn/ui, TanStack Query)
   │  (same-origin fetch; cookies are httpOnly)
   ▼
Next.js on Vercel
   ├─ Route Handlers /api/auth/*      → set/clear httpOnly session cookie (login/logout/otp)
   └─ Route Handler  /api/v2/[...path] → BFF proxy: inject `Authorization: Bearer <cookie token>`
                                          + `X-Branch-Id` + `X-Device-Id`, forward to fcms-api
   │  (server-to-server HTTPS, token from cookie)
   ▼
fcms-api (Laravel, PostgreSQL)
   ├─ EXISTING: /auth, /company, /branches, /devices, /sync, /master-items,
   │            /roles, /accounts, /*-assignments, /licenses, /app-version
   └─ NEW (this project): /api/v2/<resource> REST CRUD + /api/v2/reports/*
        • read:  active rows for company+branch (COALESCE(is_deleted,0)<>1), joined by *_sync_id
        • write: shared SyncWrite contract (sets sync_id, updated_at UTC, soft-delete, device_id=web)
        • reports: SQLite report SQL ported to PostgreSQL dialect (the big lift)
   ▼
PostgreSQL (Models/Sync/* tables — SAME tables the mobile sync push/pull use)
```

**Consistency contract:** Web and mobile share the PostgreSQL sync tables. Web reads them directly (REST); mobile reaches them via sync pull. Web writes follow §5.1 so mobile pulls them. No second source of truth.

## 5. Cross-cutting foundations (read before any phase)

### 5.1 Backend write contract — the sync invariants (load-bearing)
Every REST **write** to a sync table MUST go through a shared `SyncWriter` service that enforces, per CLAUDE.md and `SyncController.php`:
- **`sync_id`**: new UUID on create; the public REST identifier (see §5.2). Server PK `id` is also a UUID but is internal.
- **`company_id`** from the authenticated account; **`branch_id`** from the active branch (header/cookie, validated against branch-account access); **`device_id`** = the **web logical device** id (§5.3).
- **`updated_at`**: UTC `Y-m-d H:i:s` (string), set on every create/update/soft-delete. `created_at` UTC on insert. (Matches `now()->toDateTimeString()` — never ISO-serialize datetime columns; `SyncModel` keeps `$timestamps=false` and no date `$casts`.)
- **Soft-delete only**: `UPDATE … SET is_deleted=1, updated_at=<utc now>`. Never `DELETE FROM` on sync tables (hard delete allowed only on excluded tables).
- **Schema parity / `$fillable`:** any column written must be in the model's `$fillable` (Laravel silently drops non-fillable attributes — see memory "Sync field needs server `$fillable`"). All write columns already exist (they're the SyncController `allowedFields`); reuse those lists.
- **Business/event dates are LOCAL, audit/sync timestamps UTC** (§5.4 timezone). `adjustment_date`, `sale_date`, `in_spoilage_date`, purchase/order/ending-inventory dates, `payment_date` = local; `updated_at/synced_at/created_at/date_created/date_saved/input_date` = UTC.
- **Entity-specific invariants reuse the SAME code as `/sync/push`:** refactor `master_items` dedup (`MasterItemDedupKey`, SKU suffix regen) and `batch_transfer_groups` status monotonicity (`batchTransferStatusRank`) out of `SyncController` into `app/Services/Sync/` so REST writes can't bypass them.

> Net effect: a web-created row is indistinguishable from a synced row. Mobile pulls it (its `device_id` ≠ web device id, so pull's echo-suppression doesn't hide it; `updated_at > since` ships it).

### 5.2 Resource identity = `sync_id`; relationships by `*_sync_id`
- The REST API exposes **`sync_id` as the entity `id`**. Relationships are the server's `*_sync_id` columns (e.g. `items.category_sync_id` → `categories.sync_id`, `inventory_logs.item_sync_id` → `items.sync_id`).
- **Canonical schema/field catalog = `SyncController::GROUP_A`** (`fcms-api/src/app/Http/Controllers/SyncController.php:59-106`). Each entry is `entity => [Model, allowedFields]`. Use these field lists verbatim for REST serialization/validation. (Memory "Local id == sync_id": the RN client's local `id` equals `sync_id`; server stores both — always relate by `sync_id`.)
- Reads exclude soft-deleted: `WHERE company_id=? AND <branch scope> AND COALESCE(is_deleted,0) <> 1` (mirror the RN `active_<table>` views, now as Postgres views — see §5.5).
- **Branch scoping** mirrors `SyncController::pull`: `master_items` = company-wide; `batch_transfer_*` = dual-branch (source OR destination, drafts only to initiator); everything else = `branch_id = active`.

### 5.3 Branch/device model on web
- **One logical web device per company.** On first authenticated web session, register via existing `POST /api/v2/devices/register` with a synthetic stable `physical_device_id` (e.g. `web:<company_id>`), `device_name="Web (Browser)"`. Persist returned `device_id`/`device_token` in the server-side session (cookie). This `device_id` stamps all web writes.
- **Licensing:** the web device must be allowed. Either (a) add it to the license allowlist via `POST /api/v2/licenses/add-device`, or (b) add a `is_web` exemption on the device so it doesn't consume a seat. **Decide in Phase 1**; default to (b) exemption to avoid burning device caps. Web still honors company entitlement read from `GET /api/v2/licenses/me` for feature gating.
- **Branch switcher:** active branch stored in an httpOnly cookie (`fcms_branch`). The BFF injects it as `X-Branch-Id`. Switching branch = set cookie + invalidate all branch-scoped TanStack queries. Branch list from `GET /api/v2/branches`; access filtered by `branch-account-assignments` for sub-accounts (root sees all). No per-device single-branch binding for web reads — REST scopes by the chosen branch.

### 5.5 SQLite → PostgreSQL dialect port guide (reusable for every query port)
> **DB engine = PostgreSQL 16.** docker-compose runs `postgres:16-alpine`; `.env.production` has `DB_CONNECTION=pgsql`. The stale `fcms-api/.env.example` (`DB_CONNECTION=mysql`) and some `SyncController` datetime comments name a different engine — **ignore them**; dev (compose) and prod both run Postgres. Local API URL: `http://localhost:8080` (nginx `8080:80`).

The RN logic in `/src/localDbQueries/*` is SQLite. When porting to Laravel/Postgres services:
- **`IFNULL` does NOT exist in Postgres → use `COALESCE`.** The single most common port error. The RN SQL uses `IFNULL(...)` pervasively (views, cost math, report pivots); every one becomes `COALESCE(...)`.
- **Date cast:** SQLite `DATE(x)` → Postgres `x::date` (or `CAST(x AS date)`). There is no bare `DATE()` function.
- **Date math:** `DATE(x,'start of month')` → `date_trunc('month', x)::date`; month end (`'start of month','+1 month','-1 day'`) → `(date_trunc('month', x) + interval '1 month' - interval '1 day')::date`; previous-month end → `(date_trunc('month', x) - interval '1 day')::date`; `'-1 day'` → `x - interval '1 day'`. `strftime('%m %Y', x)` → `to_char(x, 'MM YYYY')`.
- **Now:** `datetime('now')`/`CURRENT_TIMESTAMP` (UTC) → `now() at time zone 'UTC'`, or write the string from PHP via `now()->toDateTimeString()` (what `SyncWriter` does). `datetime('now','localtime')` → company-tz now: `now() at time zone <company_tz>` (§5.4).
- **Soft-delete views:** create Postgres views `active_<table>` = `SELECT * FROM <table> WHERE COALESCE(is_deleted, 0) <> 1` (NULL-safe; pulled rows may have `is_deleted = NULL`). Mirror RN `createViews()`. All report/read SQL selects from these.
- **Booleans / flags:** verify each flag column's actual type in the migration (`is_deleted`, `voided`, `confirmed`, …). Integer/smallint → `<> 1` is correct; boolean → compare to `true`/`false`. Check per column before porting a comparison.
- **Relationship columns:** server uses `*_sync_id`; RN SQL joins on local `id`. When porting, join `active_inventory_logs.item_sync_id = active_items.sync_id`, etc.
- **String concat** `||` works in Postgres (same as SQLite) — no change. `NULLIF`, `CASE WHEN` identical. The `%LIKE`/`%IN` filter builder (`/src/utils/localDbHelpers.js`) → Eloquent query-builder equivalents (`ILIKE` for case-insensitive search).
- **Numeric division:** ensure cost/qty columns are `numeric`/`double precision` so `(added - removed) / NULLIF(qty,0)` is real division, not integer division (cast `::numeric` if needed).
- **avg unit cost fallback** (items): `(added_cost_net - removed_cost_net)/NULLIF(added_qty - removed_qty, 0)` wrapped in `COALESCE(..., unit_cost / (COALESCE(rate_percentage,0)/100.0 + 1))` — port verbatim (CLAUDE.md Batch Transfer; `getItems` in `/src/localDbQueries/items.js`).
- **Golden-master:** the RN reports have a golden-master test (`__tests__/reportsSqlBuilders.test.js`). For the port, build an equivalence fixture: identical seed data in RN SQLite and web Postgres → assert identical report numbers. This is how we prove "same logic."

### 5.6 Auth / session / BFF design
**Backend (additive routes in `src/routes/api.php`):** add a `Route::prefix('v2')->middleware(['auth:sanctum','throttle:api'])` group of REST resources + a `reports` prefix + a `bootstrap`/`units`/`web-settings` set. `userManagement`-gated resources reuse the existing `permission:` middleware. CORS: add `config/cors.php` allowing the Vercel origin as **defense-in-depth** (the BFF makes it unnecessary for normal flow, but enables direct calls/debug).

**Frontend BFF:**
- `app/api/auth/signin/route.ts` (+ sub-signin, signup, request-otp, verify-otp): call fcms-api auth, on success set httpOnly `fcms_session` cookie (the Sanctum token), `Secure`, `SameSite=Lax`, `HttpOnly`, plus `fcms_device`/`fcms_branch` cookies as resolved. `logout` revokes server-side (`POST /auth/logout`) and clears cookies.
- `app/api/v2/[...path]/route.ts` (Node runtime): generic proxy. Reads token from cookie → `Authorization: Bearer`; injects `X-Device-Id` (web device) and `X-Branch-Id`; forwards method/body/query to `${FCMS_API_URL}/api/v2/<path>`; streams JSON back. Centralizes 401 handling (clear cookie → client redirect to sign-in). Mirrors RN's `onUnauthorized` (`/src/api/cloudApiV2.js`).
- `middleware.ts`: gate `(app)/*` on presence of session cookie (redirect to `/sign-in`); gate on branch presence (redirect to `/branch-setup` when no `fcms_branch`).
- TanStack Query: client hooks fetch same-origin `/api/v2/*`. **QueryClient config to mirror RN** (`/src/queryClient.js`): `staleTime 5m`, `gcTime 10m` (v5 renames `cacheTime`→`gcTime`), `retry 1`, `refetchOnWindowFocus false`. Invalidate by **array keys** (memory: bare-string invalidation matches ALL in v4; in v5 keys are arrays — always use `['resource', ...]`).

## 9. Risks & gotchas (carry forward)
1. **Report SQL port** is the largest risk — prove with the equivalence test, not by eyeballing. Date-function and `active_`-view translation are where bugs hide.
2. **`$fillable` silent drops** — any new write column must be in the model `$fillable` (memory).
3. **UTC vs local datetime discipline** — business dates local (company tz), audit UTC; never ISO-serialize datetime columns on sync models (would break mobile string comparisons).
4. **Branch scoping & echo** — replicate `pull()` scoping exactly (master_items company-wide, batch_transfer dual-branch + draft-to-initiator).
5. **Non-synced data** (UOM, settings, operations seed, numbering, timezone) — easy to forget; handled in Phase 0/§5.4.
6. **Item resolution in transfers** — must use SKU/name fallbacks, never master-only, or duplicates spawn (Phase 8).
7. **TanStack Query v5** key/`gcTime` differences vs the RN v4 patterns; always array keys.
8. **Docker overlay** — server changes need a rebuild to test; plan iterations accordingly.
