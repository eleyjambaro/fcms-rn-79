# Production-Ready Environment Handling (Dev + Prod)

## Context

Today, environment handling in this app is partly wired but not safe for shipping:

- A single committed `.env` ([.env:1](.env#L1)) holds a developer's LAN IP and is loaded by `react-native-config` ([src/api/cloudApiV2.js:2-5](src/api/cloudApiV2.js#L2-L5)). Only one variable is exposed via env (`CLOUD_API_V2_BASE_URL`).
- Dev-vs-prod branching is done with `export const env = __DEV__ ? 'dev' : 'prod'` plus a comment "change to 'prod' manually to test production environment" ([src/constants/appConfig.js:7](src/constants/appConfig.js#L7)). That commented-out manual flip is the kind of thing that gets shipped by accident.
- Secrets that should be env-driven are hardcoded in source: two MongoDB Data API keys forked on `env` ([src/keys/index.js:5-8](src/keys/index.js#L5-L8)).
- The Android upload keystore is committed and the keystore password is in plaintext in a tracked file ([android/gradle.properties:42-45](android/gradle.properties#L42-L45) + `git ls-files | grep keystore` returns `android/app/debug.keystore` — and history likely has `my-upload-key.keystore`).
- The Android network security config permits cleartext globally for every domain ([android/app/src/main/res/xml/network_security_config.xml:3](android/app/src/main/res/xml/network_security_config.xml#L3)) — required for the dev LAN backend, but should not be on in release.
- A stale prod URL is hardcoded in [src/constants/urls.js:2-3](src/constants/urls.js#L2-L3) and version-check hits a hardcoded GitHub Pages URL with an unfilled iOS store URL placeholder ([src/services/versionCheck.js:8](src/services/versionCheck.js#L8), [src/services/versionCheck.js:23](src/services/versionCheck.js#L23)).

**Goal:** make dev and prod builds clearly separated, env-driven, and safe to commit — without changing app behavior beyond what's needed. Scope is limited to env handling per the user's selection; ProGuard/R8, centralized logging, and crash reporting are explicitly out of scope.

**Strategy:** keep `react-native-config` (already installed, native loading, works with Hermes + new architecture). Use its multi-env support via the `ENVFILE` environment variable. Commit only `.env.*.example` templates; real `.env.development` and `.env.production` are untracked. Add a single typed entry point that reads from `Config`, validates required vars at startup, and replaces the `__DEV__`-based env constant and the hardcoded forks.

## Plan

- Delete tracked `.env` (`git rm --cached .env`, then re-add to local working tree as untracked).
- Add `.env.development.example` and `.env.production.example` (tracked). These document the full schema for each environment.
- Add `.env.development` and `.env.production` (untracked, generated by the developer from the examples).
- Keep `.env.example` as a top-level reference pointing at both (or remove it — replaced by the two new examples).

### 2. `package.json` scripts that select the env file

Replace today's single `android` / `ios` / `start` with env-prefixed variants. `react-native-config` reads `ENVFILE` on both platforms.

```json
"start": "react-native start",
"android": "ENVFILE=.env.development react-native run-android",
"android:prod": "ENVFILE=.env.production react-native run-android --mode=release",
"ios": "ENVFILE=.env.development react-native run-ios",
"ios:prod": "ENVFILE=.env.production react-native run-ios --configuration=Release",
"bundle:android:release": "ENVFILE=.env.production cd android && ./gradlew bundleRelease",
"sync-ip": "IP=$(ipconfig getifaddr en0) && sed -i '' \"s|CLOUD_API_V2_BASE_URL=.*|CLOUD_API_V2_BASE_URL=http://$IP:8080|\" .env.development && echo \"Updated CLOUD_API_V2_BASE_URL in .env.development to http://$IP:8080\""
```

`sync-ip` now targets `.env.development` (the only file with a LAN IP).

### 3. Single typed env entry point

Create `src/config/env.js` as the only module that reads `react-native-config`. Everything else imports from here. This is the single source of truth and the single place that validates required vars.

```js
// src/config/env.js
import Config from 'react-native-config';

const required = ['APP_ENV', 'CLOUD_API_V2_BASE_URL'];

function readRequired(key) {
  const v = Config[key];
  if (!v) {
    throw new Error(
      `Missing required env var: ${key}. Did you create .env.development / .env.production from the .example files and pass ENVFILE at build time?`,
    );
  }
  return v;
}

export const env = readRequired('APP_ENV'); // 'dev' | 'prod'
export const isDev = env === 'dev';
export const isProd = env === 'prod';

export const cloudApiV2BaseUrl = readRequired('CLOUD_API_V2_BASE_URL');
export const mongodbDataApiKey = Config.MONGODB_DATA_API_KEY || '';
export const versionCheckUrl =
  Config.VERSION_CHECK_URL ||
  'https://uxi-fcms-developers.github.io/app-version/version.json';
export const iosStoreUrl = Config.IOS_STORE_URL || '';
export const androidStoreUrl =
  Config.ANDROID_STORE_URL ||
  'https://play.google.com/store/apps/details?id=rocks.uxi.fcms';

// Fail fast at module load — surfaces missing vars before navigation renders.
required.forEach(readRequired);
```

Import this from `index.js` at the very top so the validation runs before any provider mounts. On a validation failure, the JS bundle throws and the red-screen tells you exactly which var is missing — far better than the current silent `undefined` baseURL.

### 6. Android signing creds out of tracked `gradle.properties`

Move secrets out of [android/gradle.properties:42-45](android/gradle.properties#L42-L45):

- Keep `android/gradle.properties` for non-secret build flags (Hermes, new arch, JVM args).
- Move `MYAPP_UPLOAD_STORE_FILE`, `MYAPP_UPLOAD_KEY_ALIAS`, `MYAPP_UPLOAD_STORE_PASSWORD`, `MYAPP_UPLOAD_KEY_PASSWORD` to `~/.gradle/gradle.properties` (per-developer, never committed). [android/app/build.gradle:98-103](android/app/build.gradle#L98-L103) already gates on `project.hasProperty('MYAPP_UPLOAD_STORE_FILE')`, so no Gradle changes are needed.
- Add `android/app/my-upload-key.keystore` to `.gitignore` (the existing `*.keystore` line already covers it, but verify it's not currently tracked: `git ls-files android/app | grep keystore`). If it is tracked, `git rm --cached android/app/my-upload-key.keystore`.
- Add a short `android/SIGNING.md` (or extend README) documenting the layout developers need to set up locally.

The committed `android/app/debug.keystore` stays — that one is intentionally shared (every RN project ships the same one) and the `!debug.keystore` gitignore exception already preserves it.

### 7. Android network security: split dev / prod

Today, [android/app/src/main/res/xml/network_security_config.xml](android/app/src/main/res/xml/network_security_config.xml) is applied to all builds and permits cleartext globally. Split it:

- **`android/app/src/main/res/xml/network_security_config.xml`** (prod): `cleartextTrafficPermitted="false"` at the base config, keep the `data.mongodb-api.com` domain config but also flip it to `false` if the Mongo Data API endpoint is HTTPS-only (it is). Keep the pinned `server_cert` and `mongodb_api_cert` trust anchors.
- **`android/app/src/debug/res/xml/network_security_config.xml`** (debug-only, override): keep `cleartextTrafficPermitted="true"` so the LAN backend at `http://192.168.x.x:8080` continues to work. This file is only merged into debug builds.

`AndroidManifest.xml` already references `@xml/network_security_config` by name; the debug variant override is picked up automatically via Android's resource merging. No manifest changes needed.

iOS is already correctly configured (`NSAllowsArbitraryLoads=false`, `NSAllowsLocalNetworking=true` in [ios/FCMS/Info.plist]) — no changes.

## Verification

1. **Build both envs:** `npm run android` (dev) loads `.env.development`, points at the LAN IP. `npm run android:prod` loads `.env.production`, points at the prod URL. Repeat for iOS.
2. **Smoke-test login:** sign in via Cloud Auth in each build — verifies `cloudApiV2BaseUrl` resolves and `Config` is wired through.
3. **Missing-var failure mode:** rename `.env.development` away and re-run `npm run android` — expect a red-screen with `Missing required env var: APP_ENV` (or `CLOUD_API_V2_BASE_URL`). Restore the file.
4. **Network security:** in a release build (`npm run android:prod`), confirm that an http:// URL is blocked. In debug, confirm the LAN backend still works.
5. **Signing:** with creds removed from tracked `gradle.properties` and moved to `~/.gradle/gradle.properties`, `cd android && ./gradlew bundleRelease` still produces a signed AAB. Without those creds, the release build is unsigned (expected — proves we're no longer leaning on the tracked file).
6. **Repo cleanliness:** `git ls-files | grep -E "\.env($|\.development$|\.production$)|my-upload-key.keystore"` returns nothing. `git ls-files | grep -E "\.env\.example|\.env\.development\.example|\.env\.production\.example"` returns the templates.
7. **No `env` accidental flip:** `git grep "manually to test production"` returns nothing — the dangerous comment is gone.
